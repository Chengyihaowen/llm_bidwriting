# Dify 通用投标文件自动生成工作流

> 核心原则：一切格式、结构、内容要求均从招标文件中提取，不做任何预设。

---

## 工作流架构

```
开始（上传招标文件 + 填写公司信息）
  │
  ▼
Node1 [代码] 文档转文本
  │
  ▼
Node2 [LLM] 招标文件全局解析
  │
  ├──→ Node3 [LLM] 提取投标文件格式模板
  │
  ├──→ Node4 [LLM] 提取否决条款与星号条款
  │
  ▼
Node5 [条件] 用户是否上传了技术方案？
  │
  ├─ 是 → 直接使用
  └─ 否 → Node5b [LLM] 根据招标要求生成技术方案骨架，提示用户补充
  │
  ▼
Node6 [LLM] 生成技术文件（格式由 Node3 驱动）
  │
  ▼
Node7 [LLM] 技术文件自查（用 Node4 的星号条款逐条比对）
  │
  ▼
Node8 [LLM] 生成商务文件（格式由 Node3 驱动）
  │
  ▼
Node9 [LLM] 生成价格文件（格式由 Node3 驱动）
  │
  ▼
Node10 [模板] 按 Node3 提取的目录结构合并
  │
  ▼
结束 → 输出全文
```

---

## 开始节点 — 输入变量

| 变量名 | 类型 | 必填 | 说明 |
|--------|------|------|------|
| `bid_file` | File | 是 | 招标文件（.docx/.pdf） |
| `tech_file` | File | 否 | 技术方案文档，没有可不传 |
| `company_info` | Paragraph | 是 | 公司基本信息（自由文本，见下方引导语） |
| `price_info` | Paragraph | 否 | 报价信息（各分项价格或总价，没确定可不填） |

**`company_info` 填写引导语（显示在输入框占位符中）**：
```
请填写以下信息（有什么填什么，没有的留空）：
公司全称：
地址：
法定代表人：
联系人/电话/传真：
邮编：
是否由代理人投标：是/否
代理人姓名（如有）：
投标报价总价（如已确定）：
```

## Node1: 文档转文本（代码节点）

**输入**: `bid_file`, `tech_file`

```python
def main(bid_file: str, tech_file: str = "") -> dict:
    bid_text = bid_file
    tech_text = tech_file if tech_file else ""

    MAX_LEN = 120000
    if len(bid_text) > MAX_LEN:
        bid_text = bid_text[:50000] + "\n\n...[中间内容省略]...\n\n" + bid_text[-70000:]

    if len(tech_text) > MAX_LEN:
        tech_text = tech_text[:MAX_LEN]

    return {
        "bid_text": bid_text,
        "tech_text": tech_text,
        "has_tech_doc": len(tech_text.strip()) > 100
    }
```

**输出**: `bid_text`, `tech_text`, `has_tech_doc`

## Node3: 提取投标文件格式模板（LLM）

**目的**: 从招标文件中提取"投标文件应该长什么样"——目录结构、每个章节的格式模板、表格格式。这是后续所有生成节点的"模具"。

**System Prompt**:
```
你是投标文件格式提取专家。
你的任务是从招标文件中找到"投标文件格式"相关章节（通常是最后一章），
完整提取投标文件的目录结构和每个组成部分的格式模板。
这些格式模板将作为后续生成投标文件内容的严格模板，所以必须完整、准确。
```

**User Prompt**:
```
请从以下招标文件中提取投标文件的完整格式要求。

需要提取的内容：

## 1. 投标文件目录结构
（原文中规定的投标文件组成，包含所有章节名称，保持原编号）

## 2. 各部分格式模板

对于招标文件中给出的每一个格式模板（投标函、偏差表、报价表、授权委托书等），
请完整提取其格式，包括：
- 模板标题
- 表格结构（列名、行数要求）
- 填写说明/注释
- 签章要求

特别注意提取：
- 投标函的格式
- 商务偏差表的表头和列定义
- 技术偏差表的表头和列定义（精确到每一列的列名）
- 分项报价表的表头和列定义
- 资格审查资料中每个子表的格式
- 其他商务资料（承诺书、声明等）的格式

## 3. 投标文件编制要求
（纸质/电子、正副本份数、装订要求、密封要求、签字盖章要求等）

## 4. 技术文件的特殊要求
（招标文件对"技术性能指标的详细描述"有什么具体的编写指引或要求，
 比如"应逐条做出具体应答""应提供技术支持资料"等原文要求）

---

招标文件原文：

{{bid_text}}
```

**输出**: `format_template`

**User Prompt**:
```
请从以下招标文件中提取全部否决性条款和星号条款。

| 编号 | 否决条件描述（原文） | 所在章节 |

## C类：实质性响应要求

找出招标文件中定义的"实质性要求和条件"的完整清单。

## 统计
- A类星号条款数量：

---

招标文件原文：

```

**输出**: `risk_clauses`

## Node5: 条件分支

**类型**: IF/ELSE

**条件**: `has_tech_doc == true`

- **true** → Node6（使用用户上传的技术方案）
- **false** → Node5b（让 LLM 先生成技术方案骨架）

**目的**: 用户没有技术方案时，根据招标需求生成一个框架，同时告知用户哪些信息需要补充。

**System Prompt**:
```

设计一个合理的技术方案骨架。
注意：
1. 你不知道投标人是什么公司、有什么能力，所以方案要通用合理
2. 硬件选型如有具体参数要求，选择满足要求的市面主流产品
3. 对于需要投标人提供具体能力证明的部分，标注【需投标人补充：XXX】
4. 不编造投标人不存在的产品或专利
```

**User Prompt**:
```
用户没有提供技术方案文档。请根据以下招标技术要求设计技术方案骨架。

【招标文件解析——技术/供货要求部分】
{{parsed_bid}}

【星号/否决条款】
{{risk_clauses}}

（根据需求提出合理的架构方案）
```

## Node6: 生成技术文件（LLM）

**这是核心节点——它的格式完全由 Node3 的输出驱动，内容由 Node2+Node4 驱动。**

**System Prompt**:
```
你是一个专业的投标技术文件撰写专家，适用于任何行业。
你的写作必须遵循以下铁律：

1. 格式铁律：投标文件的格式（目录结构、表格列名、章节编排）
   必须严格按照招标文件规定的格式模板，不能自己发明格式。
   格式模板在【投标文件格式要求】中给出。

2. 内容铁律：技术/供货要求中的每一条都必须在投标文件中有对应响应，
   不能遗漏任何一条。需求清单在【技术/供货要求】中给出。

3. 星号铁律：星号/关键条款必须重点展开，详细描述技术方案、
   实现方式、设计原理。星号条款在【星号与否决条款】中给出。
   星号条款不允许负偏差。

4. 语言铁律：使用"我方""本项目""我公司"等第一人称表述。
   语言专业、简洁、准确。不使用"投标人"这种第三人称。

5. 评分铁律：技术评分标准中的每个得分点都要有对应内容覆盖。
   评分标准在【评标办法】中给出。
```

**User Prompt**:
```
请根据以下信息生成投标文件中的【技术文件】全部内容。

====== 输入信息 ======

【投标文件格式要求——技术文件部分】
（以下是从招标文件中提取的技术文件格式模板，你必须严格遵循此格式）
{{format_template}}

【招标文件解析——技术/供货要求】
（以下是招标文件的全部技术要求，你必须逐条响应）
{{parsed_bid}}

【星号与否决条款】
（以下条款是重点，星号条款必须详细展开，不允许负偏差）
{{risk_clauses}}

【评标办法——技术评分标准】
（你的技术文件需要覆盖以下每个得分点）
从 parsed_bid 的评标办法部分提取

【我方技术方案/能力资料】
{{tech_text}}
（如果为空，则使用：{{tech_plan_skeleton}}）

【投标人信息】
{{company_info}}

====== 生成要求 ======

请按照【投标文件格式要求】中规定的技术文件结构，生成完整内容。
具体来说，招标文件的技术文件通常包括（但以格式模板为准）：

第一部分：偏差表
- 按照格式模板中规定的偏差表表头格式生成
- 将技术/供货要求的每一条都作为一行
- 逐条填写我方的响应内容和偏差说明
- 星号条款在最后一列标注对应的技术描述章节位置

第二部分：技术性能指标的详细描述
- 按照招标文件对此部分的编写要求来组织内容
- 结构应对应技术/供货要求的结构（保持相同的编号体系）
- 星号条款展开到设计原理、算法、实现方式的深度
- 非星号条款也要逐条响应，可简洁但不能缺失
- 如果技术评分标准中有"方案完整性""技术先进性""实施方案"等评分项，
  确保有对应章节的内容

第三部分：其他技术资料
- 如果招标文件格式模板中有"其他技术资料"章节
- 根据实际情况填写，如无则填"无"

====== 输出格式 ======

直接输出 Markdown 格式的完整技术文件内容。
每个部分之间用分页符 `---` 分隔。
需要插入图片的位置标注：【此处插入：XXX图/表】
需要盖章的位置标注：【盖单位章】
```

**输出**: `tech_doc`

```
{{tech_doc}}

## 核对结果

| 条款编号 | 条款内容摘要 | 是否在技术文件中响应 | 响应位置/章节 | 响应是否充分 |
|---------|------------|-------------------|-------------|------------|

## 问题清单
（列出所有"未响应"或"响应不充分"的条款）

## 结论
- 星号条款覆盖率：___/___
```

## 模型选择建议

| 节点 | 推荐模型 | 选择理由 |
|------|---------|---------|
| Node2 全局解析 | 大上下文模型（Claude Sonnet 200K / GPT-4o 128K） | 招标文件通常很长 |
| Node3 格式提取 | 同上 | 需要读完整文件 |
| Node4 星号提取 | 同上 | 精确性要求高 |
| Node5b 方案骨架 | 通用模型即可 | 生成框架，不需要最强 |
| Node6 技术文件 | **最强模型**（Claude Opus / GPT-4o） | 最核心的内容生成 |
| Node7 自查 | 通用模型 | 比对检查，逻辑能力即可 |
| Node8 商务文件 | 通用模型 | 格式化填充为主 |
| Node9 价格文件 | 通用模型 | 简单表格 |

## 部署注意事项

### 1. 上下文长度管理

招标文件通常 3-10 万字。如果超过模型上下文限制：

**方案A：Dify 知识库**
- 将招标文件上传知识库，Node2-4 改为"检索+LLM"模式
- 优点：不受上下文限制
- 缺点：可能检索不全

**方案B：分段处理**
- Node1 中按章节切分
- 招标公告+投标人须知 → 一段
- 评标办法 → 一段
- 合同条款 → 一段
- 供货/技术要求 → 一段
- 投标文件格式 → 一段
- 各段分别送入不同LLM节点处理

### 2. Dify 迭代节点的使用

Node6→Node7 的"检查不通过则重新生成"逻辑，在 Dify 中可以用：
- **迭代节点**（Iteration）：设置最大迭代次数 2-3 次
- 或者简化为：Node7 直接输出补充建议，让用户手动决定是否需要修改

### 3. 输出后处理建议

告知用户，LLM 输出后仍需人工完成：
1. 替换所有【盖单位章】【签字】位置
2. 插入所有【此处附：XXX】标注的扫描件
3. 插入【此处插入：XXX图】标注的图片
4. 核对自查报告中的问题清单
5. 校对金额计算
6. 调整Word排版格式
7. 打印装订
